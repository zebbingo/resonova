## 快速总览

**核心结论：MQTT 协议信令层覆盖很全（95%）。硬件层除麦克风/喇叭物理特性外，其余（按键/NFC/LED/电池）均可通过前端 UI + API 响应来模拟。对于语音指令拦截测试，差距非常小。**

## 二、硬件层 ✅ 大部分可通过前端 + API 模拟

这些硬件行为在 real device 上是物理组件，但在测试场景下可以用 **前端 UI 响应 + API 注入** 来覆盖全套逻辑验证。

| # | 真实设备特性 | 模拟方式 | 状态 |
|:-:|:------------|:---------|:----:|
| 1 | **物理麦克风**：模拟信号 → ADC → I2S | ⚠️ 无损 WAV 文件替代 | 无法模拟噪声，**支撑调试够用** |
| 2 | **物理喇叭**：DAC → 功放 | ⚠️ audio_data 累积在内存 | 无法模拟播放效果，**支撑调试够用** |
| 3 | **NFC 读卡器**：读取 NFC 标签 | ✅ 前端可发不同 nfc_id，后端 API 已支持 | 前端加 NCF标签选择器即可 |
| 4 | **物理按键**：上一首/下一首/音量 | ✅ 前端可显示命令面板，支持手动发送 `response/command` | 加 UI 按钮 |
| 5 | **LED / 灯效**：状态指示、氛围灯 | ✅ `response/command` 中已含 `set_light_brightness`/`flash_light` 命令，前端可解析并画灯 | 前端加灯动画 |
| 6 | **电池/电源管理** | ✅ `state/reported` 可带电池电量字段，前端可显示电池图标 | 模拟端加字段 |
| 7 | **WiFi 模组** | ⚠️ paho 库代替 ESP-IDF 协议栈 | 协议兼容，**网络延迟差异见第五节** |
| 8 | **flash/存储** | ⚠️ identity.json | 功能一致，**数据格式差异可忽略** |

**关键结论：** 除了真实的麦克风/喇叭物理声学特性（噪声/回声/频响），**所有交互行为都可以在前端 UI 层以可视化的方式覆盖验证。** 你可以通过前端按钮触发 NFC 变更、显示 LED 状态、发送物理按键指令，完全覆盖端到端逻辑。

## 三、音频层 ⚠️ 中等差距

| # | 真实设备 | 模拟 | 影响 |
|:-:|:---------|:----|:-----|
| 1 | **Mic 音频质量**：含环境噪声、回声、变距、音量波动 | ✅ 无损 WAV/Sox 生成 | STT 准确率差异（STT 在真实音频上可能更差） |
| 2 | **AEC（回声消除）**：喇叭发出的 TTS 会被麦克风拾取 | ❌ 无 | 真实环境有回声问题，模拟测不到 |
| 3 | **VAD 精度**：真实 VAD 基于环境噪声 | ⚠️ 服务器端 VAD（Silero） | 模拟测的是服务端 VAD 而非设备 VAD |
| 4 | **编码器实现差异**：ESP32 opus 编码 vs Python opus | ⚠️ 编码器参数一致 | 微小差异，可忽略 |
| 5 | **音量可控**：真实设备有物理音量键 | ✅ 命令层模拟了 volume_up/down | 命令可以测 |

### 对语音指令测试的影响

**如果你只是测 "next/previous/pause 能不能拦截到"，音频层差距基本不影响。** 如果上线后发现 STT 准确率低于模拟，那是因为真实麦克风的音频质量——不是代码逻辑的问题。

## 四、状态机与生命周期 ✅ 接近完整

| 状态 | 模拟 | 真实设备 |
|:-----|:----:|:--------:|
| `OFFLINE → ONLINE` | ✅ power_on() → meta/online | ✅ WiFi 连上 → MQTT 连上 |
| `IDLE → CAPTURING` | ✅ session/start → 服务器 pin | ✅ 同 |
| `CAPTURING → THINKING` | ✅ audio/eos → 服务器处理 | ✅ 同 |
| `THINKING → PLAYING` | ✅ start 收到 → 设为 PLAYING | ✅ 解码播放 |
| `PLAYING → DONE` | ✅ audio/eos 收到 | ✅ 播放完 |
| 会话结束时清理 | ✅ session/end 发布 | ✅ 同 |

### 🔴 2 个差距

| # | 差距 | 说明 |
|:-:|:-----|:-----|
| 1 | **模拟不模拟 TTS 播放耗时**（收到 eos 立即发 done） | 真实设备要等播放完才发 done。这会略微影响服务器的时序调度 |
| 2 | **cue 音频收到后不真正播放**（只是回调 on_cue_audio） | 不影响主链路 |

## 五、网络与可靠性 ⚠️ 中等差距

| # | 真实设备 | 模拟 | 影响 |
|:-:|:---------|:----|:-----|
| 1 | **WiFi 连接耗时 2~5s** | MQTT 连接 <100ms | 会话建立时间不可比 |
| 2 | **WiFi 断连处理**：自动重连 + session 恢复 | ⚠️ paho 自动重连，但不恢复 session | 重连行为差异 |
| 3 | **MQTT 断线**：LWT → broker 发 offline | ✅ LWT 已设 | 基本一致 |
| 4 | **网络延迟**：100~500ms RTT（云端） | <1ms（本地 NanoMQ，⚠️ Mosquitto 已弃用） | **关键差距**——本地测的延迟远好于真实 |
| 5 | **多 device 并发**：几十万设备同时在线 | ⚠️ 最多 10 个并发 | 服务器压力测不到 |

### 对语音指令测试的影响

延迟差距会影响你测 **"从说话到收到命令的完整时间"**。
- 本地模拟：~3s 总耗时
- 真实环境：可能需要 ~5-8s（含 WiFi + 云端网络）

**但指令拦截正确性不受影响。**

## ⚠️ 2026-05-29 实测定更新：执行层阻断性问题

> 经实际全链路测试发现，上述理论覆盖度分析忽略了两个执行层阻断性问题，导致端到端链路**实际未打通**。

### 实测发现的问题

| # | 问题 | 严重度 | 发现方式 |
|:-:|:-----|:------:|:---------|
| 1 | **模拟引擎竞态条件**：`start_simulation()` 在设备正在模拟时收到新请求，后台线程因 `"Device already has an active session"` 静默失败 | 🔴 CRITICAL | 服务器日志排查 |
| 2 | **chatbot 不回响应**：即使音频 chunk 成功发送到 MQTT broker，`wait_for_turn_response()` 始终 90s 超时，STT/TTS 全部为空 | 🔴 CRITICAL | test result 分析 |
| 3 | **错误不可见**：后台线程异常被 `except` 捕获仅记日志，用户通过 API 看到的是 200 OK，实际模拟未执行 | 🔴 CRITICAL | 代码审查 |

### 对"模拟测试充分性"判定的修正

之前结论为"95% 协议层覆盖，可支撑指令拦截测试"。实际测试后发现：

**尽管协议层设计覆盖充分，但执行层的两个阻断问题导致模拟流程无法正常完成。** 在修复之前，"全链路测试"不可行。

### 修复后需重测

1. **修复竞态条件** → 验证连续 simulate 请求正常
2. **切换 VAD 模式为 EOS 触发或降低 VAD 阈值** → 验证 chatbot STT/TTS 管道正常响应
3. **重新跑 `full_link_test.py`** → 确认 end-to-end 通过

### 技术细节（2026-05-29 诊断确认）

- chatbot 使用 `bot_mqtt.py`（PID 361840，独立进程）处理 MQTT 音频，**非** `bot_runner.py`
- 管道默认运行在 VAD 模式（`MQTT_HANDLE_VAD_ON_SERVER=true`）
- SileroVADAnalyzer（默认置信度 0.8）**从未检测到语音**，导致 STT 不被触发
- 所有 33 个线程均处于 S (sleeping) 空闲等待状态
- 直接修复实验：`export MQTT_HANDLE_VAD_ON_SERVER=false` 可绕过 VAD，
  改为由 EOS 消息直接触发 STT 处理
